## 7. Возникшие проблемы и решения

### Основная проблема: Подключение к PostgreSQL

**Суть проблемы:**

1. **`host.docker.internal` не работает на Linux:**
   - На Linux этот хост не поддерживается по умолчанию (работает только на Windows/Mac)
   - Docker резолвил его в `172.17.0.1` (Docker bridge), но PostgreSQL на хосте не принимал подключения с этого адреса
   - Ошибка: `Connection refused`

2. **Внешний PostgreSQL на хосте недоступен из контейнера:**
   - PostgreSQL слушал только `localhost`, а не все интерфейсы
   - `pg_hba.conf` не был настроен для разрешения подключений из Docker сети
   - Firewall мог блокировать подключения
   - Даже при использовании `PG_HOST=144.31.87.154` подключение не проходило

3. **Переменные окружения из `.env` не загружались:**
   - В `docker-compose.prod.yml` не было `env_file: .env`
   - Переменные не попадали в контейнер

**Решение:**

Добавлен PostgreSQL контейнер в Docker:
- PostgreSQL запускается в контейнере `flask-postgres`
- Подключение через имя сервиса `postgres` внутри Docker сети
- Оба контейнера в одной сети `app-network`
- `depends_on` с `healthcheck` гарантирует готовность PostgreSQL перед запуском приложения

**Результат:**
- Подключение работает через Docker DNS (`postgres` → `172.18.0.2`)
- Не нужны настройки firewall или `pg_hba.conf` на хосте
- Приложение успешно подключается: логи показывают "Загружено контактов: 0"
- Приложение доступно по `http://144.31.87.154:5000` с HTTP статусом 200

**Дополнительные исправления:**
- Исправлена команда Gunicorn: `app.web_app:app` вместо `app.web_app:create_app()`
- Исправлен volume mapping: директория `logs` вместо файла `app.log`
- Добавлен `env_file: .env` для загрузки переменных окружения
- Исправлена ошибка с `start_time` в `after_request`
